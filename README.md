## Overview

MediFlow is a production-grade clinical analytics platform built on real-world FHIR (Fast Healthcare Interoperability Resources) data generated by [Synthea](https://synthetichealth.github.io/synthea/) — the same format used in production healthcare systems across the world.

The platform ingests raw FHIR JSON bundles, parses and flattens them into structured CSVs, loads them into Snowflake via AWS S3, and transforms them through a full medallion architecture using dbt — ultimately answering five real clinical business questions through analytics-ready mart models.

## Pipeline Stages

### Stage 1 — FHIR Parsing (`fhir_to_csv.py`)
Parses raw Synthea FHIR JSON bundles and flattens nested resources into 7 structured CSV files:
- `patients.csv`
- `encounters.csv`
- `conditions.csv`
- `medications.csv`
- `procedures.csv`
- `observations.csv`
- `claims.csv`

### Stage 2 — S3 Loading (`s3_loader.py`)
Uploads the 7 CSVs to an AWS S3 bucket, partitioned by load date.

### Stage 3 — Snowflake Ingestion
Uses Snowflake's `COPY INTO` command to load CSVs from S3 into the RAW schema.

### Stage 4 — staging Layer (dbt Views)
7 dbt views sitting on top of raw tables that standardise column names, trim whitespace, cast data types, and handle nulls — zero storage cost and using **merge strategy** that load only new records on each run

### Stage 5 — intermediate Layer (dbt Incremental)
Ephemeral models that join and enrich staging tables. Being ephemeral, Deduplication is handled via `unique_key`.

### Stage 6 — mart layer (dbt Ephemeral)
5 analytics-ready mart models each answering a specific clinical business question.

## dbt Models

### staging Layer — Views
Lightweight views with no storage cost. Each view:
- Renames raw Synthea columns to readable snake_case names
- Casts string columns to correct data types (`TIMESTAMP_NTZ`, `NUMBER`, `VARCHAR`)
- Trims whitespace and removes special characters
- Applies `REGEXP_REPLACE` to strip numeric suffixes from name fields
- Performes a merge incremental laod 

Each staging model uses:
```sql
{{ config(
    materialized         = 'incremental',
    incremental_strategy = 'merge',
    unique_key           = '<natural_key>',
    on_schema_change     = 'append_new_columns'
) }}
```
The `incremental_filter` macro drives the watermark:
```sql
{% macro incremental_filter(timestamp_col) %}
  {% if is_incremental() %}
    where {{ timestamp_col }} > (select max({{ timestamp_col }}) from {{ this }})
  {% endif %}
{% endmacro %}
```
- First run → full load
- Subsequent runs → only new records since last max timestamp
- Safe to re-run — no duplicates

### Intermediate Layer — Ephemeral (Joins)
Ephemeral models compile directly into downstream mart CTEs:
- `int_condition_patients` — enriches every condition with patient demographics
- `int_encounter_claims` — links every claim back to its clinical encounter

### Mart Layer — Tables
Full table refreshes on each run, built on top of staging and intermediate models.

## Custom Macros

All macros are written from scratch — no reliance on `dbt_utils` for core logic.

| Macro | Purpose | Example Output |
|-------|---------|---------------|
| `incremental_filter(col)` | Watermark WHERE clause for incremental loads | `WHERE col > (SELECT MAX(col) FROM this)` |
| `surrogate_key(columns)` | MD5 hash of multiple columns into a unique ID | `MD5(COALESCE(CAST(col AS VARCHAR), 'NULL') \|\| '-' \|\| ...)` |
| `age_in_years(birthdate)` | Precise age calculation accounting for leap years | `FLOOR(DATEDIFF('day', birthdate, CURRENT_DATE()) / 365.25)` |
| `age_bucket(age_col)` | Groups age into labelled bands | `CASE WHEN age < 18 THEN '0-17' ...` |
| `re_admission(date, patient, days)` | LAG-based 30-day readmission flag | `CASE WHEN DATEDIFF('day', LAG(date) OVER(...), date) <= 30 ...` |
| `days_between(start, end)` | Null-safe day difference | `DATEDIFF('day', start, end)` |
| `safe_divide(num, denom)` | Division with zero protection | `DIV0(numerator, denominator)` |
| `nullif_zero(col)` | Returns NULL if value is zero | `NULLIF(col, 0)` |
| `round_cost(col, decimals)` | Standardises cost column decimal places | `ROUND(col, 2)` |

## Mart Models — Clinical Questions Answered

### Mart 1 — Patient Readmissions
**Question:** Which patients were readmitted within 30 days of a previous inpatient visit, and what does that tell us about care quality?"

Uses `LAG()` window function partitioned by `PATIENT_ID` to look back at each patient's previous encounter date. The `re_admission` macro flags any inpatient encounter occurring within 30 days of the previous one.

**Key columns:** `PREVIOUS_VISIT`, `DAYS_SINCE_LAST_VSIST`

---

### Mart 2 — Condition Prevalence
**Question:** What are the most common diagnoses by age group, gender, and race?

Joins `condition_stage` to `patient_stage` and aggregates by demographic dimensions. Calculates prevalence rate per 1,000 patients using `safe_divide`.

**Key columns:** `PATIENTS_WITH_CONDITION`, `TOTAL_GROUP_PATIENTS`

---

### Mart 3 — Average Length of Stay
**Question:** How long do patients stay per encounter type and diagnosis?

Calculates length of stay in **minutes** (not days) since most Synthea encounters are same-day. Joins `encounter_stage` to `condition_stage` and aggregates `AVG(LOS_MINUTES)` per encounter type and condition.

**Key columns:** `LENGTH_OF_STAY`, `MODE_STAY`, `MEDIAN_STAY`

---

### Mart 4 — Medication Patterns
**Question:** What are the most prescribed medications per condition?

Joins `medication_request_stage` to `condition_stage` on both `ENCOUNTER_ID` and `PATIENT_ID` — the dual join key prevents cartesian products for patients with multiple visits.

**Key columns:** `TOTAL_PRESCRIPTIONS`, `PATIENTS_COUNT`

## Setup & Installation

### Prerequisites
- Python 3.9+
- Snowflake Account
- AWS S3 Bucket with programmatic access credentials
- dbt Core 1.11+


### 1. Clone the repository
```bash
git clone https://github.com/DavidAsikpo/mediflow-clinical-analytics.git
cd mediflow-clinical-analytics
```

### 2. Create and activate a virtual environment
```bash
python -m venv venv
source venv/bin/activate        # Mac/Linux
venv\Scripts\activate           # Windows
```

### 3. Install dependencies
```bash
pip install -r requirements.txt
```

### 4. Set environment variables

Create a `.env` file in the root of your project — **never commit this file to GitHub**:

```bash
AWS_ACCESS_KEY_ID="your_aws_key"
AWS_SECRET_ACCESS_KEY="your_aws_secret"
BUCKET_NAME="your-mediflow-bucket-name"
SF_USER="your_snowflake_user"
SF_PASSWORD="your_snowflake_password"
SF_ACCOUNT="your_snowflake_account_locator"
SF_WAREHOUSE="your_snowflake_warehouse"
```

Make sure `.env` is listed in your `.gitignore`:
```bash
echo ".env" >> .gitignore
```

### 5. Configure dbt profile

Add the following to `~/.dbt/profiles.yml`:

```yaml
mediflow:
  target: dev
  outputs:
    dev:
      type: snowflake
      account: "{{ env_var('SF_ACCOUNT') }}"
      user: "{{ env_var('SF_USER') }}"
      password: "{{ env_var('SF_PASSWORD') }}"
      role: <your_role>
      database: MEDIFLOW
      warehouse: "{{ env_var('SF_WAREHOUSE') }}"
      schema: RAW
      threads: 4
```

### 6. Test your dbt connection
```bash
dbt debug
```

### 7. Run the pipeline
```bash
python orchestrator.py
```

## Key Engineering Decisions

**Incremental over full refresh for staging** — staging tables can grow to hundreds of thousands of rows. Incremental merge ensures only new data is processed on each run, reducing Snowflake compute costs and run times significantly.

**Ephemeral intermediate models** — intermediate joins are needed by multiple mart models but don't need to be stored in Snowflake. Ephemeral materialisation inlines them as CTEs, saving storage costs while keeping the logic reusable.

**Custom macros over hardcoded SQL** — all repetitive SQL patterns (surrogate keys, incremental filters, age calculations, cost rounding) are abstracted into macros. A single change to a macro propagates across every model that uses it.

**Dual key joins for medication-condition** — joining medications to conditions on both `ENCOUNTER_ID` and `PATIENT_ID` prevents cartesian products for patients with multiple encounters — a common data modelling pitfall in clinical datasets.

**Surrogate keys on all mart models** — every mart row has an MD5-based surrogate key generated from its natural dimension columns. This enables dbt uniqueness tests, supports future incremental materialisation of marts, and provides stable join keys for BI tools.

**`DATEDIFF` in minutes for LOS** — Synthea encounter durations are typically same-day, making day-based length of stay calculations return zero for most records. Minutes provides meaningful granularity across all encounter classes.
